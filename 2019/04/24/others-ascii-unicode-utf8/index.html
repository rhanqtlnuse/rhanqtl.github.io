字符集：ASCII、Unicode 和 UTF-8

Publicado el 2019-04-24

最近想实现一个自己的 C++ string 类，对 string 类编码问题早有耳闻（当然也亲身体验过），所以打算先学习一下字符编码相关的知识。

ASCII 码

ASCII 码是比较早（不是最早）的一种字符编码，最初占 8 位（计算机中最小的寻址单元）只支持 128 个字符（字母只支持英文字母），最高位被规定为 0

扩展 ASCII 码

使用 ASCII 码会导致一个问题：法语中字母上方有注音符号，德语有变音字母，这些都无法使用 ASCII 码表示，所以后来出现了 “扩展 ASCII 码”，即在原来 ASCII 码的基础上，利用了剩余的 128 个位置，编入了新的符号。比如，法语中的 é 的编码为 130（二进制 10000010）。这样一来，这些欧洲国家使用的编码体系，可以表示最多 256 个符号。

但是，问题又来了。不同的国家有不同的字母，因此，哪怕它们都使用 256 个符号的编码方式，代表的字母却不一样。比如，130 在法语编码中代表了 é，在希伯来语编码中却代表了字母 Gimel（ג），在俄语编码中又会代表另一个符号。

至于亚洲国家的文字，使用的符号就更多了，汉字就多达 10 万左右。一个字节只能表示 256 种符号，肯定是不够的，就必须使用多个字节表达一个符号。比如，简体中文常见的编码方式是 GB2312（对 ASCII 的中文扩展），使用两个字节表示一个汉字，所以理论上最多可以表示 256 x 256 = 65536 个符号。

Unicode

为了解决各地（国）编码不统一的问题，ISO 提出了 Universal Multiple-Octet Coded Character set，简称 UCS，俗称 Unicode。这个编码使用两个字节表示一个字符，可以容纳地球上所有文字和符号的编码。

这样，原来的 strlen 函数就出现了错误 —— 本来，一个汉字相当于 2 个英文字符，但是在 Unicode 中，一个汉字就是一个字符。

Unicode 规定使用 2 字节表示一个字符，所以对于 ASCII 中的半角字符，Unicode 维持其编码，只是将高字节设为全 0，这就引出了一个问题：存储空间的浪费。如果一个文本中全是 ASCII 字符，那么占用的存储空间将会有一半是浪费的！

UTF-8

Unicode 在很长一段时间内无法推广，直到互联网的出现，为解决 Unicode 如何在网络上传输的问题，于是面向传输的众多 UTF（UCS Transfer Format）标准出现了，顾名思义，UTF-8 就是每次 8 个位传输数据，而 UTF-16 就是每次 16 个位。UTF-8 就是在互联网上使用最广的一种 Unicode 的实现方式，这是为传输而设计的编码，并使编码无国界，这样就可以显示全世界上所有文化的字符了。UTF-8 最大的一个特点，就是它是一种变长的编码方式。它可以使用 1 到 4 个字节表示一个符号，根据不同的符号而变化字节长度，当字符在 ASCII 码的范围时，就用一个字节表示，保留了ASCII 字符一个字节的编码作为它的一部分（需要注意的是 Unicode 一个中文字符占 2 个字节，而 UTF-8 一个中文字符占 3 个字节）。从 Unicode 到 UTF-8 并不是直接的对应，而是要过一些算法和规则来转换：

Unicode（十六进制） UTF-8（二进制）
0000 0000 - 0000 007F 0xxxxxxx
0000 0080 - 0000 07FF 110xxxxx 10xxxxxx
0000 0800 - 0000 FFFF 1110xxxx 10xxxxxx 10xxxxxx
0001 0000 - 0010 FFFF 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

需要注意的是，Unicode 是字符集，而 UTF-8 是编码规则。字符集为每一个字符分配一个唯一的 ID（学名为码位/码点/code point），编码规则是将 code point 转换为字节序列的规则。

字符的存储

通常，在内存中使用 Unicode 来表示字符，而使用 UTF 来存储字符，这样：

  • 取字符时只需要总是取 2 个字节即可，节省了处理（计算）的时间，提高处理效率
  • 存储/传输字符时可以节省空间/带宽，提高传输效率

其中，Unicode 称为程序 “内码”，UTF 称为程序 “外码”。
