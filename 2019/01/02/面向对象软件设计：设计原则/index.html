面向对象软件设计：设计原则

发表于 2019-01-02 | 分类于 Design ， OOP | 浏览 次

设计原则综述

  • 软件的可维护性与复用性
    • 面向对象软件设计复用的目标在于实现支持可维护性的复用，而这要以面向对象软件设计原则为基础，这些设计原则首先都是复用的原则，遵循这些设计原则可以有效地提高系统的复用性，同时提高系统的可维护性
    • 面向对象设计原则也是对系统进行合理重构的指南针

单一职责原则

  • 定义
    • 第一种表述：一个对象应该只包含单一的职责，并且该职责被完整地封装在一个类中
    • 第二种表述：就一个类而言，应该仅有一个引起它变化的原因
  • 分析
    • 一个类（模块、方法）承担的职责越多，它被复用的可能性越小，而且耦合度也会过高
    • 类的职责主要表现在两个方面：数据职责和行为职责。数据职责通过属性来体现，行为职责通过方法来体现
    • 单一职责原则是实现高内聚、低耦合的指导方针

  • 首先，登陆模块的作用是验证用户和密码，返回验证的结果，所以 validate() 方法体现了这个类的职责

  • findUser(...) 和 getConnection()：这个工作是 Data 层的工作

根据以上的分析，我们应该留下 init() display() validate() 方法，将其他的方法（职责）交给其他的类

再看，其实 getConnection() 方法也可以从 DataService 中分离出来，因为一个有一定规模的系统中，一定不止有一个 DataService，而所有的 DataService 都需要连接数据库。

开闭原则

一个软件实体应该对扩展开放，对修改关闭。也就是说在设计一个模块的时候应该使这个模块能够在不改变内部代码/设计的情况下改变这个模块的行为（扩展）。

  • 软件实体可以指一个软件模块、一个由多个类组成的局部结构或一个独立的类

  • 关键是 “抽象化”

  • 更具体的原则：对可变性封装原则，即找到系统中的可变因素，并将其封装

里氏替换原则

使用基类的地方应该都能够使用相应的子类代替，而无需了解子类的细节（透明），反之不成立。

所以在程序中尽量使用基类类型来对对象进行定义，而在运行时再确定其子类类型，用子类对象来替换父类对象。

接口隔离原则

针对接口编程，不要针对实现编程

  • “接口” 并不是某个语言中的语法特性（比如 Java 的 interface）而是合约（方法）

与依赖倒置原则的关系

一个经典的例子：Java 中的 Stack 类继承自 Vector

但其实仔细想想，Stack 是一个（is a）Vector 吗？

并不是，Stack 只是部分地使用了 Vector 的功能，所以更合理的方式应该是 Stack 持有一个（has a）Vector 的引用，如下：

再进一步思考，其实 Stack 也并不是非要使用 Vector，其实它需要的只是一个顺序的容器，能够支持在一端增/删/查即可。所以可以进一步地修改，令 Vector 继承自 List，Stack 持有 List 的引用，这样我们甚至可以在运行时动态地修改 Stack 所依赖的顺序容器，而不会影响 Stack 的正常使用。这样的设计提供了更大的灵活性，同时也只给 Stack 提供了必要的接口（这里也有接口隔离原则）。

将 Stack 由继承自 Vector 改为持有 Vector 的引用，遵循了合成复用原则；而将 Stack 由持有 Vector（具体类）的引用改为持有 List（抽象接口）的引用，遵循了依赖倒置原则。
