并查集

分析

这里分析使用 weighted union 和 path compression find 的情况。

迭代对数

$H(i) = 2^{H(i-1)}$，并且 $ H(0) = 1$

i 0 1 2 3 4 5
$H(i)$ 1 2 4 16 65536 $2^{65536}$

可以看到，这个函数增长的很快（宇宙中原子个数有 $10^{80}$ 个，而 $2^{65536}$ 比 $10^{80}$ 还要大，所以通常情况下 $i$ 最多为 5）。我们要使用的函数是其反函数：

$log^*(n) = \min{ i | H(i) \ge n } = \min{ i | \log_2\log_2\cdots\log_2n \le 1 }$，其中 $\log$ 有 $i$ 次，“迭代对数”的名字也因此而来。

结点的分组

我们依据 $H(i)$（或 $log^*(n)$）将结点分为如下的组：

组号 0 1 2 3 4 5
所在层数 0 1 2 ~ 3 4 ~ 15 16 ~ 65535 65536 ~ $2^{65536}-1$

其中，第 0 组只有根结点，第 1 组是根结点的直接子结点。

下面是来自维基百科的一幅图，将每个分组看作一个桶（Bucket）：

对于某一次 Find 操作 $A_i$，将结点到根的路径上的所有结点作为一个集合 $X_i$，将集合 $X_i$ 分为如下三组：

$Y_i = { \mu \in X_i | \mu 为根或根的直接子结点 }$

$Z_i = { \mu \in X_i - Y_i | g(\mu) \lt g(p_i(\mu)) }$

$W_i = { \mu \in X_i - Y_i | g(\mu) = g(p_i(\mu)) }$

某次 Find 操作的代价是上面三个集合中每个点的代价的总和。

那么我们可以得到如下的分析：

  • 对于 $Y_i$，虽然每次可能要换桶，但是由于此集合中是根结点或者根结点的直接子结点，我们可以保证换桶的次数在常数时间内，故时间复杂度为 $O(m)$；
  • 对于 $Z_i$，对于一棵有 n 个结点的树，最多有 $\log^(n)$ 个不同的组，所以某条路径上能够属于 $Z_i$ 的结点最多有 $\log^(n)$ 个，那么换桶的次数最多为 $\log^(n)$ 次，故时间复杂度为 $O(m\log^(n))$；
  • 对于 $W_i$，不需要换桶，所以时间复杂度正比于其中点的个数。

下面有两个引理来辅助 $W_i$ 中点的个数的计算：

  • Level Census Lemma：第 $l$ 层最多有 $n / 2^{l-1}$ 个结点
    • 证明：假设第 $l$ 层有 $p$ 个结点，这些结点共有 $s$ 棵子树。我们知道，高度为 $h$ 的树（这里的树是 Union-Find 树）至少有 $2^h$ 个结点，那么高度为 $l-1$ 的 $s$ 棵子树共有 $s \cdot 2^{l-1}$ 个结点，同时必有 $s \cdot 2^{l-1} \le n$，也就是 $s \le n/2^{l-1}$，又 $p \le s$，所以 $p \le n/2^{l-1}$。
  • Group Census Lemma：编号为 $g$ 的组中最多有 $2n/H(g)$ 个结点
    • 证明：编号为 $g$ 的组包括树的第 $H(g-1) + 1$ 到第 $H(g)$ 层，由上面的 Level Census 引理，我们有：$\sum_{l=H(g-1)+1}^{H(g)} \dfrac{n}{2^{l-1}} = \dfrac{n}{2^{H(g-1)}} + \dfrac{n}{2^{H(g-1)}\cdot 2} + \cdots + \dfrac{n}{2^{H(g-1)}\cdot 2^{H(g)-H(g-1)}}\le \dfrac{n\cdot(1 + \frac{1}{2} + \cdots)}{2^{H(g-1)}} = \dfrac{2n}{H(g)}$

同时，我们知道同一个桶中，某个结点改变其父结点的次数最多为 $H(g) - H(g-1) - 1​$ 次。接下来我们就可以计算 $W_i​$ 中结点总数：

综上，总的时间复杂度为 $O(m) + O(m\log^(n)) + O(2n(1+\log^(n))​$，由于我们假设 $m \ge n​$，所以最终得到使用带权合并和路径压缩的 Union-Find 程序的时间复杂度为 $O(m\log^*(n))​$

P 和 NP

P：考虑一个决策问题 P，I 是 P 的一个实例，令 |I| 表示 I 在二进制编码下的长度。如果对于所有的 I，P 的一个算法 A 的运行时间为 (|I|)$^{O(1)}$，那么 A 就是多项式边界的。

  • P 是有多项式边界的算法的决策问题的集合
  • P 是可以在多项式时间内解决的决策问题的集合
  • 注意：在集合 P 中不代表有高效的算法，比如 $\Theta(n^{100})​$
  • 不在 P 中的问题解决代价很高，P 中的问题有闭包属性，在 P 中的性质独立于计算模型

NP

NP（Non-deterministic Polynomial Time）：非确定性图灵机。如果对于问题 P 的每一个实例，NTM 在 T(|I|) 步内正确地返回了 yes 或 no，我们就说 NTM 在 T(n) 时间内解决一个决策问题 P

  • NP 是所有能够被 NTM 在多项式时间内验证的问题的集合。

红黑树

发表于 2018-12-04 | 分类于 算法 (Algorithms) ， 树 (Tree)

什么是红黑树？

红黑树首先是一棵二叉搜索树，具有几乎完美的平衡性质

叙述性定义

红黑树是这样一棵二叉搜索树：

  • 根结点为黑色
  • 每个内部结点要么是黑色，要么是红色
  • 叶结点（NIL）为黑色
  • 颜色约束：红色结点不能有红色子结点（换言之，红色结点只能有黑色子结点）
  • 黑色高度约束：从某结点出发到其所有后代叶结点的简单路径上，均包含相同数目的黑色结点

递归定义

  • ARB（Almost-RB）树：除根结点为红色之外，满足 RB 树其他约束
  • 定义 RB$_0$ 为空树（或外部结点），以 $h$ 表示树的黑高度
  • RB$_{h+1}$：子树可以是 RB$_h$ 或 ARB$_{h+1}$
  • ARB$_{h+1}$：子树均为 RB$_h$

为什么要使用红黑树？

C++ 中的 map

我们知道 HashMap 的平均时间复杂度为 O(1)（可以参考《算法导论》的相关分析），map 的平均时间复杂度为 $O(\lg n)$ 的，实践上是不是 HashMap 一定优于 map 呢？这里面有几个因素要考虑：

  1. HashMap 内存效率比 map 差，这是显而易见的；
  2. map 的查找效率实践上是非常高的，如在 1M 数据中查找一个元素，仅需要 20次 比较；
  3. map 的查找效率比 HashMap 稳定，hash 最坏的情况为 $O(n)$（个人观点：无论使用什么 hash 函数，总能够找到最坏情况的键序列，虽然可以使用全域哈希的方法，但也并不是完全能够避免最坏情况，而且 hash 函数一不小心就会…，相比之下，红黑树的 $O(\lg n)$ 很稳，而且在可以接受的范围内）；
  4. HashMap 查找时候要算 hash，这个最坏时间复杂度是 $O(m)$（M是key字符串的长度），如果你的 key 非常非常非常非常非常非常……长，基于比较的 map 通常只使用头几个字符进行比较，而 HashMap 要 $O(M)$ 地算出 hash；
  5. 内存布局会影响内存局部性，对性能会有影响。

二分搜索

如果函数 f(x) 是单调的，并且在已知区间 [a, b] 中存在某个值 c 使得 f(c) 等于给定的值 d，那么可以使用二分法来求出 c。

红黑树

（注意高度与黑高度的不同！）

定义

一个二叉搜索树，每个节点或为红色，或为黑色（所有的外部节点都为黑色），满足如下约束：

  • 颜色约束：红色的结点不能有红色的子结点
  • 黑高度约束：给定节点 u 的所有外部路径的黑色长度都相同
  • 根结点是黑色的

ARB：根结点为红色，满足颜色约束和黑高度约束

递归定义

  • 基本情况：RB$_0$ 为外部结点
  • ARB$_h$（h >= 1）
    • 根节点为红色
    • 两棵子树都为 RB$_{h-1}$
  • RB$_h$（h >= 1）
    • 根结点为黑色
    • 子树可以是 RB$_{h-1}$ 或 ARB$_h$

性质

  • RB$_h$ 或 ARB$_h$ 的黑高度是良定义的，为 h
    • “良定义” 是指根结点的所有外部路径的黑色长度都相同
  • RB$_h$
    • 至少有 2$^h$ - 1 个黑色内部结点
    • 至多有 4$^h$ - 1 个内部结点
    • 任何一个黑色结点的深度至多是树的黑色深度的两倍
  • ARB$_h$
    • 至少有 2$^h$ - 2 个黑色内部结点
    • 至多有 4$^h$ / 2 - 1 个内部结点
    • 任何一个黑色结点的深度至多是树的黑色深度的两倍
  • 有 n 个内部结点的红黑树的黑色深度至多为 log(n+1)，树的高度至多为 2 log(n+1)

操作

插入

思路：类似二叉搜索树，保持黑高度约束（将新插入的结点设为红色），调整树以符合颜色约束

4 - 结点簇
3 - 结点簇

四种情况

插入之后可能有两种情况：

  • 插入之后的子树既不是 RB 也不是 ARB
  • 插入之后的子树是 ARB，但树的根结点是红色的

下面用五个常量来表示插入之后的状态。

插入之后树的顶端颜色状态
  • ok：不需要调整
  • rbr：插入之后的树为 RB
  • brb：插入之后的树为 ARB，如果父结点的颜色为红色，需要调整（此时从父结点的角度来看，树的顶端颜色状态为 rrb 或 brr）
  • rrb：插入之后的树的根结点为红色，左子树是 ARB，右子树是 RB（此时父结点一定是黑色的）。要根据父结点的情况来确定是 3-结点簇还是 4-结点簇
  • brr：插入之后的树的根结点为红色，左子树是 RB，右子树是 ARB。调整方法与 rrb 类似

删除

思路：类似二叉搜索树，保持颜色约束，调整树以符合黑高度约束

  • 如果删除之后，黑高度不变，不需要调整，删除完成
  • 如果删除之后，子树的黑高度减 1，需要调整
